Hallazgo
Raíz dual del árbol causa–efecto

Las cinco fallas comparten una sola raíz dual: gobernanza ausente e instrumentación ausente se retroalimentan.

Raíz A — Gobernanza

Sin plan SQA formal, sin roles, sin criterios de aceptación, sin función QA independiente. Nadie decide qué es "lo suficientemente bueno" para producción.

Raíz B — Instrumentación

Sin métricas en producción, sin monitoreo del modelo ML, sin pipeline CI/CD que ejecute verificaciones automáticas en cada commit.

Implicación técnica

Sin gobernanza no se decide qué instrumentar; sin datos de instrumentación no se puede gobernar. Las dos raíces se retroalimentan: atacar solo una garantiza recaída.

Decisión a validar: los tres entregables intervienen ambas raíces de forma simultánea, no consecutiva.

Hallazgo central — el más importante de la presentación. Si ustedes recuerdan una sola cosa que sea esta: las cinco fallas tienen una raíz dual, no una raíz simple.

Raíz A — gobernanza: nadie decide qué es calidad aceptable. Raíz B — instrumentación: nadie mide qué tan bien está funcionando el sistema. PAUSA

El bucle es perverso: sin gobernanza, no se prioriza instrumentar nada; y sin métricas, la gerencia no sabe qué decisiones de gobernanza tomar. Por eso un equipo puede pasarse años "mejorando calidad" sin moverse del lugar: ataca un lado del bucle, no los dos.

Pipeline CI/CD/SQA. Once etapas que convierten cada commit en un punto de control automático. El principio es Shift Left Testing: mover las verificaciones lo más temprano posible para reducir el costo de corregir.

Las etapas bloqueantes son cuatro. PAUSA

Primera, SAST con SonarQube: ningún bug crítico pasa al siguiente stage. Segunda, pruebas unitarias con cobertura mínima 80% — si el código nuevo no tiene suficientes pruebas, el pipeline se detiene.

Tercera, la que no existe en pipelines convencionales: el ML Quality Gate. Cuando hay cambios en el módulo de recomendaciones, el sistema evalúa Recall, Precision y drift antes de aprobar el despliegue. Punto.

Cuarta, canary release: el despliegue comienza con 10% del tráfico y solo se completa si las métricas igualan al baseline durante 30 minutos. Si no, rollback automático.

Si preguntan por velocidad: el pipeline completo es menor a 30 minutos. Las etapas costosas — pruebas de carga — no corren en cada PR, corren semanalmente o ante cambios arquitectónicos.

Entregable 3 · SQA-AI
De aseguramiento reactivo a predictivo

Cuatro componentes de IA convierten el SQA en un sistema que predice, genera y monitorea — no solo detecta.

1 · Defect Prediction Engine

XGBoost (algoritmo de árboles de decisión potenciados) entrenado con el historial de bugs. Predice probabilidad de defecto por archivo modificado.

Lessmann et al. (2008): AUC 0,70–0,90 en proyectos reales.

2 · LLM Test Case Generator

Para archivos con probabilidad > 0,70, genera tests vía API de Claude con el diff (cambio entre versiones) como contexto y restricciones de cobertura explícitas.

3 · ML Quality Monitor

Evidently AI monitorea Recall@10, Precision@10 y drift en tiempo real. Si detecta degradación, dispara alerta y agenda reentrenamiento.

4 · NLP Documentation Analyzer

Analiza semánticamente la documentación. Detecta módulos sin documentar o textos inconsistentes con el código. Ataca directamente el diagnóstico D5.

Lógica del Orchestrator

El campo overall_passed es una conjunción lógica (AND): si el ML Quality Gate falla, el pipeline falla completo aunque las pruebas unitarias estén perfectas. Es diseño intencional, no descuido.

Por qué empezar por el componente 3: el riesgo de mayor impacto y menor controlabilidad con pruebas convencionales es el modelo ML en producción.

Entregable 3 — SQA-AI. La propuesta más avanzada. Convierte el aseguramiento de reactivo — detectar fallos después — a predictivo — anticipar dónde van a ocurrir.

Cuatro componentes. PAUSA

1. Defect Prediction Engine: XGBoost entrenado con el historial de bugs del repositorio. Califica cada archivo modificado con una probabilidad de introducir un defecto. Usa características como complejidad ciclomática, número de autores y bugs históricos en ese módulo.

2. LLM Test Case Generator: para archivos de alto riesgo (probabilidad > 0,70), llama a la API de Claude con el diff y genera tests focalizados en valores límite y caminos alternativos.

3. ML Quality Monitor: Evidently AI monitorea el modelo de recomendaciones en producción. Si detecta degradación o drift, alerta y agenda reentrenamiento.

4. NLP Documentation Analyzer: analiza la documentación con NLP para detectar módulos sin documentar o textos inconsistentes con el código. Ataca el D5.

Detalle de diseño: el campo overall_passed del orchestrator es un AND, no un OR. Si el modelo ML falla, el pipeline falla completo. Intencional.

Entregable 3 · Pseudocódigo
Clase 1 de 4 — DefectPredictionEngine

Convierte cada archivo modificado en un vector numérico y devuelve la probabilidad de que introduzca un defecto.

defect_prediction_engine.py Python · pseudocode 
class DefectPredictionEngine:
    def __init__(self, model_path: str):
        self.model = load_model(model_path)   # XGBoost entrenado con bugs

    def extract_features(self, file_diff: FileDiff) -> FeatureVector:
        return FeatureVector(
            lines_added       = file_diff.additions,
            lines_removed     = file_diff.deletions,
            cyclomatic_complex= compute_cyclomatic(file_diff.new_content),
            halstead_volume   = compute_halstead(file_diff.new_content),
            num_authors       = count_unique_authors(file_diff.git_history),
            prior_bug_count   = get_bug_count(file_diff.path, lookback_days=180),
            days_since_change = file_diff.staleness_days
        )

    def predict(self, files: List[FileDiff]) -> List[DefectScore]:
        return [DefectScore(file=f, prob=self.model.predict(self.extract_features(f)))
                for f in files]
Qué hace

Recibe los archivos modificados en un commit y le entrega al sistema cuáles tienen alta probabilidad de introducir un defecto.

Las 7 características

Líneas +/−, complejidad ciclomática (caminos), volumen de Halstead (densidad lógica), nº de autores, bugs previos, días sin tocar.

Umbral de decisión

Probabilidad > 0,70 → archivo de alto riesgo → activa la generación automática de pruebas (clase 2).

Por qué XGBoost: Lessmann et al. (2008) reporta AUC 0,70–0,90 con esta familia de modelos sobre características de proceso y código.

Primera clase del Orchestrator: DefectPredictionEngine. Su trabajo es decirle al pipeline qué archivos del commit son sospechosos antes de probarlos.

Funciona en dos pasos. PAUSA Primero, extract_features convierte cada archivo en un vector de siete números — líneas añadidas y eliminadas, complejidad ciclomática (cuántos caminos independientes tiene el código), volumen de Halstead (densidad lógica), cuántos autores distintos han tocado ese archivo, cuántos bugs previos hubo en ese módulo en los últimos seis meses, y hace cuántos días fue modificado por última vez.

Segundo, predict le pasa ese vector al modelo XGBoost y obtiene una probabilidad entre 0 y 1. Si la probabilidad supera 0,70, el archivo se marca como de alto riesgo.

Si preguntan por qué XGBoost: es un algoritmo de árboles potenciados, robusto con pocos datos y features heterogéneas. Lessmann et al. (2008) reporta AUC entre 0,70 y 0,90 en proyectos reales.

Clase 2 de 4 — LLMTestCaseGenerator

Para cada archivo de alto riesgo, llama a la API de Claude con el diff y produce tests ejecutables, no genéricos.

llm_test_case_generator.py Python · pseudocode 
class LLMTestCaseGenerator:
    def __init__(self, api_key: str):
        self.client = AnthropicClient(api_key=api_key)

    def generate(self, file: FileDiff, score: DefectScore) -> List[TestCase]:
        prompt = f'''
            Archivo: {file.path}  |  Riesgo: {score.prob:.2f}
            Framework de pruebas: {file.test_framework}
            Diff del archivo:
            {file.diff_text[:2000]}   # truncado para contexto

            Genera casos de prueba unitarios para los cambios del diff.
            Prioriza: valores límite, condiciones de error, paths alternativos.
            Formato: {file.test_framework} válido, listo para ejecutar.
            Solo el código de prueba, sin explicaciones adicionales.
        '''
        response = self.client.messages.create(
            model='claude-sonnet-4-20250514',
            max_tokens=2000,
            messages=[{'role': 'user', 'content': prompt}]
        )
        return parse_test_cases(response.content[0].text, file.test_framework)
Anatomía del prompt

3 datos concretos al LLM: ruta del archivo, riesgo calculado y diff truncado a 2.000 caracteres. Sin esa estructura, la salida sería genérica.

Restricciones explícitas

El prompt obliga a priorizar valores límite, errores y caminos alternativos, en el framework exacto del proyecto.

Parseo

parse_test_cases extrae el código de la respuesta y lo convierte en objetos TestCase ejecutables.

Diseño defensivo: los tests para módulos críticos (pagos, recomendaciones) pasan por revisión humana antes de incorporarse como regresión permanente — el "test oracle problem".

Segunda clase: LLMTestCaseGenerator. Usa la API de Claude para generar pruebas focalizadas en los archivos que la primera clase marcó como riesgosos.

La lógica del método generate es importante: no le pide al LLM tests genéricos. Le entrega tres cosas concretas como contexto — ruta del archivo, riesgo calculado, y el diff real del cambio, truncado a 2.000 caracteres para no exceder el contexto. PAUSA

Y le da instrucciones específicas: priorizar valores límite, condiciones de error y caminos alternativos, en el framework de pruebas exacto del proyecto. El resultado se parsea y se convierte en objetos TestCase listos para ejecutarse.

Si el profesor pregunta si no es peligroso confiar en un LLM para tests: sí, y se llama "problema del test oracle". El LLM puede generar pruebas sintácticamente correctas pero con resultado esperado incorrecto. Por eso los tests para módulos críticos requieren revisión humana antes de regresión permanente. El LLM reduce el tiempo de escritura, no reemplaza el juicio sobre qué probar.

Entregable 3 · Pseudocódigo
Clase 3 de 4 — MLQualityMonitor

Mide el modelo en producción con cuatro métricas y bloquea el despliegue si alguna cae bajo el umbral.

ml_quality_monitor.py Python · pseudocode 
class MLQualityMonitor:
    def __init__(self, thresholds: QualityThresholds):
        self.thresholds = thresholds   # recall=0.85, precision=0.80, ndcg=0.75
        self.evidently  = EvidentlyMonitor()
        self.mlflow     = MLflowClient()

    def evaluate(self, new_model, production_model, holdout_data) -> MLQualityReport:
        metrics = {
            'recall_10':    recall_at_k(new_model, holdout_data, k=10),
            'precision_10': precision_at_k(new_model, holdout_data, k=10),
            'ndcg_10':      ndcg_at_k(new_model, holdout_data, k=10),
            'drift_score':  self.evidently.compute_drift(new_model, production_model)
        }
        self.mlflow.log_metrics(metrics)
        passed = all([
            metrics['recall_10']    >= self.thresholds.recall,
            metrics['precision_10'] >= self.thresholds.precision,
            metrics['ndcg_10']      >= self.thresholds.ndcg,
            metrics['drift_score']  <= self.thresholds.max_drift
        ])
        if not passed:
            AlertingSystem.send(channel='sqa-ml-alerts',
                message=f'ML Quality Gate FAILED: {metrics}', severity='HIGH')
        return MLQualityReport(metrics=metrics, passed=passed)
4 métricas evaluadas

Recall@10, Precision@10, NDCG@10 (ranking) y drift (deriva entre nuevo modelo y el de producción).

Umbrales

recall ≥ 0,85 · precision ≥ 0,80 · ndcg ≥ 0,75 · drift ≤ máx. Una sola métrica fuera → passed = False.

Tercera clase: MLQualityMonitor. Esta es la que resuelve el problema más crítico de TourVista: hoy nadie sabe qué tan bien funciona el modelo en producción.

El método evaluate calcula tres métricas estándar de sistemas de recomendación — Recall@10, Precision@10 y NDCG@10, que miden qué tan bien el modelo recupera los paquetes relevantes dentro de las diez primeras recomendaciones. Si preguntan qué es NDCG: Normalized Discounted Cumulative Gain — premia que los resultados más relevantes aparezcan más arriba en la lista. Es la métrica estándar para rankings, donde no basta con acertar sino con acertar en orden.

class SQAAIOrchestrator:
    def __init__(self):
        self.dpe  = DefectPredictionEngine('models/defect_predictor_v2.pkl')
        self.ltcg = LLMTestCaseGenerator(api_key=env('ANTHROPIC_API_KEY'))
        self.mlqm = MLQualityMonitor(thresholds=DEFAULT_THRESHOLDS)

    def run(self, commit: Commit) -> PipelineQualityReport:
        # Paso 1: Extraer archivos modificados
        changed_files = commit.get_changed_files()

        # Paso 2: Predecir defectos por archivo
        scores    = self.dpe.predict(changed_files)
        high_risk = [f for f, s in zip(changed_files, scores) if s.prob >= 0.70]

        # Paso 3 + 4: Generar y ejecutar pruebas para alto riesgo
        generated_tests = []
        for file, score in zip(high_risk, [s for s in scores if s.prob >= 0.70]):
            generated_tests.extend(self.ltcg.generate(file, score))
        test_results = TestRunner.execute(generated_tests)

        # Paso 5: Evaluar modelo ML si cambió el módulo de recomendaciones
        ml_report = None
        if any('recommendations' in f.path for f in changed_files):
            ml_report = self.mlqm.evaluate(
                new_model=commit.get_artifact('recommendation_model'),
                production_model=get_production_model(),
                holdout_data=load_holdout_dataset()
            )

        # Paso 6: Consolidar reporte — overall_passed es un AND lógico
        return PipelineQualityReport(
            defect_scores   = scores,
            high_risk_count = len(high_risk),
            tests_generated = len(generated_tests),
            tests_passed    = test_results.pass_count,
            tests_failed    = test_results.fail_count,
            ml_quality      = ml_report,
            overall_passed  = test_results.all_passed() and
                              (ml_report is None or ml_report.passed)
        )
Flujo de 6 pasos

1) Archivos del commit · 2) Predecir defectos · 3) Filtrar alto riesgo · 4) Generar + ejecutar tests · 5) Evaluar ML si aplica · 6) Consolidar.

El AND intencional

overall_passed es una conjunción, no opción: si el ML Quality Gate falla, el pipeline falla completo aunque los tests pasen.

Evaluación condicional

El bloque ML solo se ejecuta si el commit toca recommendations/. Eficiencia y foco en el riesgo real.

Diseño defendible: la conjunción evita que un modelo degradado se cuele en producción solo porque las pruebas unitarias salieron verdes.

Cuarta clase: SQAAIOrchestrator. Esta es la clase principal. Su método run coordina las tres anteriores en seis pasos secuenciales.

Paso 1: extrae los archivos modificados del commit. Paso 2: el DefectPredictionEngine calcula probabilidades. Paso 3: filtra los de alto riesgo, probabilidad mayor a 0,70. Paso 4: el LLMTestCaseGenerator produce pruebas y las ejecuta. Paso 5: si el commit tocó el módulo de recomendaciones, el MLQualityMonitor evalúa la calidad del nuevo modelo. Paso 6: consolida todo. PAUSA

Detalle importante: el campo overall_passed es una conjunción lógica, no una opción. Si el modelo ML falla, el pipeline falla completo, aunque las pruebas unitarias estén perfectas. Es diseño intencional, no descuido.

Si preguntan por qué evaluar el ML condicionalmente: porque la evaluación es costosa — exige cargar el modelo en producción, el nuevo modelo y un holdout dataset. Solo tiene sentido si el commit cambió algo en recommendations/. Es eficiencia con foco en el riesgo real.

Entregable 3 · Evaluación crítica
Impacto teórico y limitaciones declaradas

El sistema baja el escape de defectos a producción de ~80% a <20% — pero exige revisión humana y métricas de diversidad.

Impacto en 4 dimensiones
  • Calidad — defect escape rate < 20% en 6 meses
  • Costo — ahorro > costo de IA al evitar 2–3 incidentes/mes
  • Mantenibilidad — completitud documental medible por sprint
  • UX — menos fallos + recomendaciones con umbral mínimo
Limitación 1 — Test oracle

El LLM puede generar tests sintácticamente correctos pero semánticamente errados. Los tests para módulos críticos (pagos, recomendaciones) exigen revisión humana antes de incorporarse como regresión permanente.

Limitación 2 — Popularity bias

Optimizar solo por accuracy genera recomendaciones precisas pero poco diversas. El monitor debe incluir intra-list diversity y coverage, no solo accuracy.

Prerequisito honesto

Si el repositorio no tiene 200–300 bugs etiquetados, hay fase de recolección 3–6 meses. Mientras tanto, complejidad ciclomática como proxy de riesgo.

Postura del trabajo: la IA reduce el tiempo de escritura de pruebas; no reemplaza el juicio sobre qué se debe probar.

Impacto y límites. Cuatro dimensiones del impacto: calidad (escape rate baja de 80% a 20%), costo (ahorro al evitar 2–3 incidentes mensuales), mantenibilidad (documentación medible por sprint), UX (menos fallos visibles).

Pero señalo dos limitaciones que el trabajo no oculta. PAUSA

Test oracle: el LLM puede generar tests con sintaxis correcta pero comportamiento esperado incorrecto si el contexto del prompt es incompleto. Por eso los tests para pagos y recomendaciones requieren revisión humana. La IA ayuda, no reemplaza al QA Engineer.

Popularity bias: si solo optimizo accuracy, el modelo recomienda siempre los destinos populares y mata la diversidad. El monitor debe incluir métricas de diversidad además de accuracy.

Si preguntan cómo entreno el modelo de predicción sin datos: esa es la limitación real. Necesita 200–300 bugs etiquetados. Si no existen, fase de recolección de 3–6 meses, y mientras tanto uso complejidad ciclomática como proxy de riesgo. No es óptimo, pero es honesto y funcional.
